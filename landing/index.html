What Falsky Detects

Six categories of flakiness, each with its own detection algorithm

Timing Flakes

Tests that fail due to timeouts, slow responses, or race conditions. Z-score anomaly detection on duration data catches tests with erratic run times that signal hidden instability.

z-score analysis

Order Dependency

Tests that pass or fail depending on execution order. Sequential pattern analysis catches alternating and periodic failures that only appear when tests run in specific sequences.

sequential patterns

Shared State

Tests that contaminate each other through shared databases, files, or services. Clustered failure patterns reveal the culprit — when multiple tests fail together, shared state is the cause.

cluster analysis

Non-Deterministic Data

Tests that fail randomly due to unstable test data, UUIDs, or timestamps. High error diversity is the signal — when error messages vary across failures, randomness is at play.

error diversity

Environment Specific

Tests that pass locally but fail in CI (or vice versa). Cross-environment pass rate comparison catches these — a test that's 95% reliable locally but 40% in CI is environment-specific.

env comparison

Bayesian Scoring

Every test gets a trust score (0–100) using Beta-Binomial conjugate priors. Better than naive pass/total calculations, especially with small sample sizes where simple ratios mislead.

beta-binomial

How It Works

Four steps from zero to full trust visibility

1

Connect your repo

Sign in with GitHub, select your repository. Falsky auto-detects your test framework — no manual configuration needed. Works with pytest, Jest, JUnit, Mocha, and anything that outputs JUnit XML.

2

Add the GitHub Action

Copy one YAML block into your workflow. The Action reads your test results and sends them to Falsky's Trust Engine after every CI run. No extra scripts, no sidecar containers. falsky-action@v1

3

Get Bayesian trust scores

Each test receives a trust score from 0 (completely unreliable) to 100 (rock solid). Flaky tests are categorized with confidence levels — timing, order, shared state, environment, or random.

4

Fix before it breaks

See trends over time, catch degrading tests early, and quarantine unreliable ones before they block your pipeline. Dashboard shows run history, category breakdowns, and actionable fix suggestions.

One YAML Block. Done.

Add this to your workflow and Falsky starts scoring immediately

.github/workflows/ci.yml
# After your test step
- uses: Pritahi/falsky-action@v1
  with:
    api_key: ${{ secrets.FALSKY_API_KEY }}
    repo: ${{ github.repository }}
    report_path: test-results.xml
